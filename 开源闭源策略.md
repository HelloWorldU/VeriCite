开源部分（GitHub）：

### **方案 B：完全闭源 + GitHub 做营销**

**GitHub 只放**：
- README（产品介绍）
- Issues（收集需求）
- Releases（下载预编译二进制）

**代码不开源**，但提供：
- 免费试用（10 次/月）
- 透明的技术博客（解释原理）
- 公开的 Benchmark（建立信任）

**案例参考**：Grammarly、Turnitin 都没开源，但在 GitHub 有官方 repo 做社区支持。

## 🧠 决策思维模型

### **1. Jobs-to-be-Done（用户雇佣产品解决什么问题）**

不是问"用户需要什么功能"，而是问：

| Job | DIY（开源） | 付费解决 |
|------|-------------|----------|
| "我需要验证 1 篇论文" | ✅ 免费跑本地 | ❌ 不值得付费 |
| "我需要验证 100 篇论文" | ⚠️ 配置复杂，耗时 | ✅ 付费批量处理 |
| "我需要集成到期刊系统" | ❌ 没技术能力 | ✅ 必须买企业版 |
| "我需要 24/7 可靠性" | ❌ 自己维护太累 | ✅ 付费 SaaS |

**结论**：开源覆盖"简单任务"，付费覆盖"专业需求"。

---

### **2. Value Chain Analysis（价值链分析）**

用户的完整工作流：
```
写论文 → 插入引用 → **验证引用** → 修改 → 提交
```

你可以在哪些环节收费？

| 环节 | 开源可做 | 付费增值 |
|------|----------|----------|
| 验证单个引用 | ✅ CLI 工具 | - |
| 批量验证 | ⚠️ 手动循环 | ✅ 一键批处理 |
| 修复建议 | ❌ 只报错 | ✅ AI 推荐正确引用 |
| 集成 Overleaf/Zotero | ❌ | ✅ 插件形式 |
| 团队共享规则 | ❌ | ✅ 协作功能 |

**关键洞察**：开源覆盖 20% 的核心功能，付费提供 80% 的便利性。

---

### **3. Christensen 的创新理论（破坏性创新）**

**当前市场**：
- 高端：Turnitin（贵，机构采购）
- 低端：手工检查（免费但慢）
- **你的机会**：中端（个人可负担 + 自动化）

**策略**：
- 开源吸引"低端用户"（学生）
- 付费服务"向上迁移"（导师、期刊）

## 🎯 推荐产品形态

**Hybrid 模型（最优解）**：
```
┌─────────────────────────────────────┐
│  GitHub 开源                         │
│  - Core 验证引擎                     │
│  - 文档 + 技术博客                   │
│  - Issues 收集需求                   │
│  → 作用：SEO + 技术信任 + 社区贡献   │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  官网 SaaS（vericite.com）           │
│  - 云服务（无需安装）                │
│  - 高级功能（批量 + 标注 + API）     │
│  - Freemium 模式                     │
│  → 作用：主要收入来源                │
└─────────────────────────────────────┘
```

**具体设计**：
- **开源版**：单文件 CLI，本地 OCR，基础验证
- **Cloud Free**：每月 10 次，网页界面
- **Cloud Pro**（$5/月）：无限次数 + 高精度 OCR + PDF 标注
- **Enterprise**（定制）：API + 批量 + SSO

**为什么这样有效**：
- 开发者/技术用户用开源版 → 在公司推荐你的付费版
- 非技术用户（教授）直接买云服务
- 机构（图书馆）看到 GitHub 活跃度 → 信任你的商业产品
